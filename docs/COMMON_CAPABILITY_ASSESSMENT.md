# Common Capability Assessment

这份文档用于收口根级 TODO 里那批“先评估是否要做”的常用能力，目标不是立刻建模块，而是先把结论定下来：哪些值得做、应该落在哪、哪些暂时不单拆。

## 评估原则

- 优先看是否已经和现有模块强耦合，能不新增模块就不新增
- 优先收敛稳定入口，不急着把供应商或中间件细节抽成过早 SPI
- 如果能力本身边界不稳定，先落文档结论，不急着建模块
- 只有在跨项目复用价值高、语义独立时，才考虑单独模块

## 结论矩阵

| 能力项 | 结论 | 建议落点 | 优先级 | 主要原因 |
| --- | --- | --- | --- | --- |
| 消息重试 / 延迟消息 / 死信治理 | 做，但先不拆独立模块 | `getboot-mq` 下新增增强子树 | 中 | 强依赖消息中间件语义，独立模块容易失真 |
| 文件上传 / 预签名 / 媒体处理 | 做，但拆成两层 | `getboot-storage` 先承接上传与预签名；媒体处理后置 | 高 | 对象存储边界稳定，媒体处理边界不稳定 |
| 第三方账号登录 / 开放平台接入 | 做，但低优先级且不并入现有 `getboot-auth` | 后续单独规划 `getboot-auth-social` 或同类模块 | 低 | OAuth 登录与令牌态鉴权不是一类问题 |
| 统一邮件发送 | 做，适合独立模块 | 后续规划 `getboot-mail` | 中 | 边界清晰，但复用频率通常低于短信与对象存储 |
| 验证码 / 人机校验 / 风控接入 | 拆开看 | 验证码可后续做轻量模块；人机校验 / 风控先保留接入点 | 中 | 这三类能力稳定性和抽象难度差异很大 |

## 逐项说明

### 1. 消息重试 / 延迟消息 / 死信治理

结论：

- 值得做
- 但先不单拆 `getboot-mq-enhancement` 之外的新模块

建议：

- 先放在 `getboot-mq` 内部继续演进
- 目录上可预留 `api.enhancement`、`support.retry`、`support.deadletter`、`infrastructure.<mq>.enhancement`

原因：

- 延迟消息、死信、重投在 RocketMQ / Kafka 上差异很大
- 如果先抽成仓库级独立模块，统一层会被迫透传大量中间件细节
- 更合理的做法是先让 `getboot-mq` 继续吸收这类增强能力，等稳定后再判断是否需要拆出单独边界

### 2. 文件上传 / 预签名 / 媒体处理

结论：

- 值得做
- 但只先做对象存储主链路，不把媒体处理一起混进第一阶段

建议：

- 先按 [`docs/MODULE_ROADMAP.md`](./MODULE_ROADMAP.md) 落 `getboot-storage`
- 第一阶段只覆盖上传、删除、下载、预签名 URL、Bucket 路由
- 媒体处理、转码、审核、CDN 域名等放到后续扩展层

原因：

- 对象存储主链路边界清楚，跨项目复用价值高
- 媒体处理往往深度依赖业务流程或云厂商，过早抽统一层收益不高

### 3. 第三方账号登录 / 开放平台接入

结论：

- 有价值
- 但不应直接塞回 `getboot-auth`

建议：

- 暂定后续评估独立边界，例如 `getboot-auth-social`
- 第一阶段不承诺统一覆盖所有 OAuth / Open Platform 差异

原因：

- `getboot-auth` 当前核心是令牌态鉴权与当前用户访问，不是第三方登录编排
- 第三方登录涉及授权跳转、回调、账户绑定、平台差异，和现有鉴权模块不是一个层级

### 4. 统一邮件发送

结论：

- 适合做
- 但排位在 `getboot-sms` 之后

建议：

- 后续单列 `getboot-mail`
- 第一阶段只收 SMTP / 常见云邮件的发送门面、模板变量和附件能力

原因：

- 邮件发送边界足够稳定，抽象难度低
- 但相对短信、存储、幂等来说，当前业务复用优先级略低

### 5. 验证码 / 人机校验 / 风控接入

结论：

- 不建议一次做成一个大而全模块

建议：

- 验证码能力可在后续单独评估轻量模块，如 `getboot-verification`
- 人机校验先保留统一 SPI 接入点，不急着做统一门面
- 风控接入继续视为业务链路更重的集成点，不急着单拆模块

原因：

- 验证码本身相对稳定
- 人机校验和风控高度依赖业务场景、厂商策略和规则配置
- 三者强行揉在一起，模块边界会很快失控

## 对路线图的影响

- 路线图首批模块 `getboot-idempotency / getboot-storage / getboot-sms / getboot-search / getboot-ai` 已完成首版落地
- `getboot-mq` 后续增强优先在模块内扩展，不新增独立消息治理模块
- `getboot-mail` 仍可作为下一批候选模块，但优先级低于已完成的首批模块
- 第三方登录与验证码相关能力，继续保留方向，不抢占下一轮高优先级模块位

## 当前路线补充

在 [`docs/MODULE_ROADMAP.md`](./MODULE_ROADMAP.md) 现有状态之外，再补一个仓库级判断：

1. 首批高频基础能力模块已经完成首版落地
2. 下一轮优先考虑 `getboot-mq` 模块内增强，而不是额外拆消息治理新模块
3. `getboot-mail`、第三方登录、验证码相关能力继续按复用价值和边界稳定性评估
