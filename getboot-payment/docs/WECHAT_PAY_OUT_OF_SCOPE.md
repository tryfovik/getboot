# 微信支付暂不纳入清单

这份文档记录“官方左侧树里存在，但当前不准备纳入 `getboot-payment` 稳定 API”的能力。

## 当前暂不纳入

- `付款码支付（V2）`
- `现金红包（V2）`
- `清关报关（V2）`
- `刷脸支付`
- `医保支付`
- `支付即服务`
- `微信支付公钥`

## 原因

### 1. 协议或安全模型不一致

- `付款码支付（V2）`
- `现金红包（V2）`
- `清关报关（V2）`

这些能力要么仍基于另一套 V2 协议面，要么和当前模块以 `v3 + 官方 Java SDK` 为主的路线不一致，不适合直接塞进现有稳定接口。

### 2. 场景依赖过强

- `刷脸支付`
- `医保支付`
- `支付即服务`

这些能力明显依赖线下机具、行业权限或较强业务流程，不适合作为基础 starter 默认承载。

### 3. 当前已有替代承载

- `微信支付公钥`

当前模块已经提供 `WechatPaySecurityService.encryptSensitiveValue()`，可以先走平台证书公钥加密路线；等确实出现“必须使用微信支付公钥产品 API”的稳定需求，再新增独立子树更合适。

## 当前结论

针对 `支付即服务 / 微信支付公钥` 是否要新增独立稳定子树，当前结论仍然是：`暂不新增`。

- `支付即服务`

这类能力通常依赖行业权限、进件关系、线下流程或代运营链路，不适合作为基础 starter 默认暴露稳定 API。若业务侧临时需要，优先通过 `WechatPayApiService` 或业务项目自定义适配承接；只有在仓库内出现稳定、可复用的请求/响应模型后，再考虑独立子树。

- `微信支付公钥`

当前 `WechatPaySecurityService` 已经覆盖“平台证书下载 + 敏感字段加密”这条主路径，足以支撑现阶段接入。只有在后续明确引入“微信支付公钥产品 API”且它的配置、验签、轮换语义能稳定抽象时，才考虑新增独立稳定子树。

## 后续原则

- 真正需要纳入时，优先新增独立子树，不混入 `PaymentService`
- 高行业属性能力优先考虑拆为独立模块，而不是继续膨胀 `getboot-payment`
- 仅当官方能力已稳定、业务抽象边界清晰时，才从 `WechatPayApiService` 升级为稳定 API
